¿Qué tipo de bucles hay en JS?

Definición

En Javascript podemos encontrar seis tipos de bucles:

  • for: es el bucle más común, funciona mediante un contador numérico y se repite de forma controlada hasta que ese contador deja de cumplir la condición que le hayamos puesto.
  • while: es un bucle que se ejecuta continuamente mientras la condición que hayamos indicado sea verdadera (true). Evalúa la condición antes de ejecutar el código cada vez.
  • do...while: es muy parecido al while, pero se diferencian en que este evalúa la condición al final de la vuelta en vez de al principio. Esto asegura que el código se ejecute al menos una vez, sin importar si la condición es verdadera o falsa desde el inicio.
  • for...in: es un bucle diseñado específicamente para recorrer las propiedades (llaves/keys) de un objeto de JS. Va saltando de propiedad en propiedad de forma automática.
  • for...of: es un bucle que sirve para recorrer os valores de elementos iterables como arrays o cadenas de texto. Su diferencia con for…in es que este no da la llave o índice, sino que da el valor real directo.
  • forEach(): aunque no es en sí un bucle como los anteriores, sino un método nativo exclusivo de los arrays de JS, funciona de manera parecida. Recibe una función callback y la ejecuta una vez por cada elemento del array.

Usos

El bucle for es comúnmente utilizado para cuando se sabe el número exacto de veces que se quiere ejecutar el código, o para recorrer arrays mediante sus índices numéricos; o incluso para crear secuencias numéricas o patrones.

El bucle while se utiliza para cuando no se sabe cuántas vueltas se van a necesitar en el bucle antes de ejecutarlo (por ejemplo, leer las líneas de un archivo hasta que se acabe).

El bucle do…while se suele utilizar para menús interactivos de consola o sistemas de validación en los que se necesita pedirle un dato al usuario obligatoriamente antes de comprobar si es correcto o no.

El bucle for…in se utiliza normalmente para inspeccionar qué datos hay guardados dentro de un objeto dinámico o para hacer configuraciones basadas en diccionarios.

El bucle for…of se usa para recorrer arrays de forma limpia cuando no importa en qué posición o índice está el elemento, sino el elemento en sí.

El método forEach() se suele usar para renderizar listas de elementos en el DOM o para aplicar operaciones rápidas a colecciones en paradigmas de programación funcional.

Ventajas y Desventajas

for

Ventajas
  • El más rápido en cuanto a rendimiento.
  • Da control absoluto sobre el índice (se puede ir de dos en dos, hacia atrás, saltarse elementos…).
Desventajas
  • La sintaxis es propensa a errores de escritura, es fácil equivocarse en la condición y crear un bucle infinito o un error de “desfase por uno”.

while

Ventajas
  • Sintaxis limpia y directa para evaluar estados booleanos.
Desventajas
  • Si se olvida actualizar la variable que altera la condición dentro del bloque, este se ejecutará para siempre, congelando el navegador (bucle infinito).

do...while

Ventajas
  • Ahorra el tener que duplicar el código antes del bucle para que se ejecute por primera vez.
Desventajas
  • Al igual que pasa con while, puede dar lugar a bucles infinitos.

for...in

Ventajas
  • Es la forma más directa y cómoda de extraer las propiedades de un objeto sin tener que usar índices numéricos.
Desventajas
  • Es muy lento en los arrays.
  • Recorre las propiedades en un orden que no siempre está garantizado, y hereda propiedades del prototipo del objeto si no se tiene cuidado.

for...of

Ventajas
  • La sintaxis es muy limpia, moderna, segura y fácil de leer.
  • No hay peligro de bucles infinitos.
Desventajas
  • No se tiene acceso al índice actual. Si se necesita modificar el array original basándose en su posición, este bucle no sirve.

forEach()

Ventajas
  • Da acceso al elemento, al índice y al array completo al mismo tiempo en los argumentos de la función.
  • Mantiene el scope (alcance de las variables) muy limpio.
Desventajas
  • No se pueden usar palabras como break ni continue. Si se inicia, se tiene que recorrer el array entero de principio a fin, no se puede parar a mitad.

Buenas prácticas

En general en todos los tipos de bucles, algunas buenas prácticas son las siguientes.

Es recomendable evitar la fatiga de memoria en arrays; si se va a usar el bucle for para recorrer un array, es mejor guardar la longitud en una variable en lugar de calcularla en cada vuelta.

También es mejor usar const en for...of y for...in, ya que, como en cada vuelta del bucle se crea una nueva variable completamente limpia en el bloque de memoria, al usar siempre const en vez de let se evitan mutaciones accidentales del dato dentro de esa vuelta.

En el JavaScript moderno, si la intención es transformar datos, se prefiere .map(), .filter(), .reduce() antes que un bucle for manual. Esto hace que el código sea más fácil de leer y probar.

Por último, hay que tener cuidado con los bucles infinitos en el desarrollo. Si se está construyendo un bucle while, hay que escribir la línea de actualización de la variable (i++ o parecido) antes de realizar el resto de la programación, así no se olvida.

Sintaxis

  • for:

    • for (inicio; condición; actualización) { //Código a ejecutar };

  • while:

    • while (condición) { // Código a ejecutar };

  • do...while:

    • do { // Código a ejecutar (se ejecuta mínimo una vez) } while (condición);

  • for...in:

    • for (const llave in objeto) { // Código a ejecutar };

  • for...of:

    • for (const elemento of iterable) { // Código a ejecutar };

  • forEach():

    • array.forEach((elemento, indice) => { // Código a ejecutar });

    Ejemplos

                        console.log(`Vuelta número: ${i}`);
                    };
                
                        let tazasDeCafe = 3;

                    while (tazasDeCafe > 0) {
                        console.log(`Te quedan ${tazasDeCafe} tazas.`);
                        tazasDeCafe--; 
                    };
                
                        let contraseñaCorrecta = false;

                    do {
                        console.log("Comprobando credenciales con la base de datos...");
                    } while (contraseñaCorrecta);
                
                        const coche = { marca: "Toyota", modelo: "Corolla", año: 2024 };

                    for (const propiedad in coche) {
                        console.log(`${propiedad}: ${coche[propiedad]}`);
                    };
                
                        const videojuegos = ["Zelda", "Mario", "Metroid"];

                    for (const juego of videojuegos) {
                        console.log(`Jugando a: ${juego}`);
                    };
                
                        const precios = [10, 20, 30];

                    precios.forEach((precio, i) => {
                        console.log(`Producto ${i + 1} cuesta: ${precio}€`);
                    });

    ¿Cuáles son las diferencias entre const, let y var?

    Definición

    Tanto const, como let como var son formas de almacenar una variable.

    var es la forma original de declarar las variables. Tiene un alcance de función (function scope), es decir, si se declara dentro de una función, la variable solo le pertenece a la función, pero si se declara en un bloque (como if o for), la variable se “escapa” y es accesible desde fuera.

    let se creó para solucionar los problemas de var. Tiene alcance de bloque (block scope), es decir, la variable solo le pertenece al bloque en el que esté creada, ya sea un if, un for, un while o una función, no se puede acceder a ella desde fuera.

    const tiene el mismo alcance de bloque que let, pero es mas restrictiva, ya que es una constante. Esto significa que, una vez asignado el valor, no se puede reescribir posteriormente. Además, es obligatorio inicializarla (darle un valor) en la misma línea en la que se declara.

    Usos

    Hoy en día, var no tiene ningún uso, se mantiene solamente por compatibilidad con páginas web muy antiguas, ya que está desaconsejado su uso.

    Entre los usos de let tenemos los contadores de los bucles (for (let i=0;…)), y la conmutabilidad de las variables, ya que, si queremos crear una variable y posteriormente cambiar su valor, tendríamos que usar let.

    const sí tiene algunos usos más, ya que permite guardar referencias a elementos del DOM (Document Object Model, const botón = document.querySelector(…)), configurar URLs de APIs, configuraciones globales, funciones de flecha y módulos importados, y almacenar las estructuras de datos (objetos y arrays) que modificarán sus datos internos pero no su tipo de estructura.

    Ventajas y Desventajas

    var

    Ventajas
    • Era muy flexible.
    Desventajas
    • Es muy peligros, permite re-declaración accidental de una variable sin que el navegador dé ningún error.
    • Su comportamiento de hoisting inicializa la variable como undefined, lo que suele ocultar bugs muy difíciles de encontrar.

    let

    Ventajas
    • Evita que las variables “contaminen” otras partes del código.
    • No permite la re-declaración en el mismo ámbito (si se intenta duplicar una variable con el mismo nombre dará error).
    • Su hoisting protege metiendo la variable en la Temporal Dead Zone (TDZ), dando error si se intenta usar antes de definirla.
    Desventajas
    • No tiene ninguna reseñable, opera de la manera esperada en cualquier lenguaje de programación moderno.

    const

    Ventajas
    • Aporta seguridad y robustez al código; si se lee const, el programador/a sabe que esa variable mantiene su identidad durante todo el código.
    • Optimiza ligeramente el rendimiento del motor de JavaScript.
    Desventajas
    • Puede confundir a los principiantes con los objetos y los arrays, ya que const protege la referencia en la memoria, no los valores internos (mutabilidad).

    Buenas Prácticas

    Una de las prácticas recomendadas es escribir const por defecto. Si se descubre que hay que reasignar una variable más abajo (por ejemplo, un contador), se cambia a let. Se estima que el 90% de las variables en una aplicación deberían ser const.

    El uso de var está completamente desaconsejado, no hay ningún escenario en el desarrollo moderno de aplicaciones en el que var sea necesario, solo introduce riesgos de seguridad. En su lugar es mejor usar let.

    Por último, es recomendable usar mayúsculas solo para las constantes matemáticas o fijas, como const PI = 3.14 o const API_URL = …, ya que son valores absolutos que nunca cambian ni mutan, así son más fáciles de identificar. Para el resto de variables hechas con const se usa la nomenclatura estándar camelCase.

    Sintaxis

  • var

    • var nombre = contenido;

  • let

    • let nombre = contenido;

  • const

    • const nombre = contenido;

    Ejemplos

                    edad = 26;
                
                        let contador =1;
                    contador = 2;

    ¿Qué es una función de flecha?

    Definición

    Una función de flecha es una alternativa compacta a las expresiones de función tradicionales, con dos características matemáticas y de comportamiento únicas:

    • Sintaxis reducida: elimina la necesidad de escribir la palabra clave function y, en muchos casos, los corchetes y la palabra return.
    • Contexto léxico de this: a diferencia de una función normal, las funciones de flecha no crean su propio valor de this. En su lugar, heredan el this del código que las rodea (su entorno de ejecución inmediato).

    Usos

    Uno de los principales usos son los callbacks y funciones anónimas, ya que ayuda a pasar funciones rápidas como argumentos dentro de métodos de arrays (map, filter, forEach).

    También permite crear temporizadores y eventos asíncronos, creándolos dentro de funciones como setTimeout o promesas, donde se necesita mantener la referencia del this del componente superior.

    Por último, permiten la declaración rápida de componentes funcionales.

    Ventajas y Desventajas

    Ventajas
    • Legibilidad extrema: limpian visualmente el código eliminando palabras repetitivas como function y return.
    • Menor problemas con this: al no crear su propio this, evitan tener que escribir this.valor = valor; o usar .bind(this) para que una función de fondo recuerde a qué objeto pertenece.
    • Ideales para programación funcional: fomentan la creación de funciones puras, pequeñas y directas.
    Desventajas
    • No sirven como métodos de objetos: si se mete una función de flecha dentro de un objeto, this no apuntará al objeto, sino a la ventana global (window), rompiendo la lógica.
    • No tienen el objeto arguments: si se necesita usar la variable oculta arguments (común en funciones antiguas), las funciones de flecha no la tienen (en su lugar se usa el operador Rest …args).
    • No se pueden usar como constructores: no se les puede poner la palabra new por delante, JS lanzará un error crítico si sucede. No tienen prototipos (prototype).

    Buenas prácticas

    Una buena práctica es mantener el retorno implícito corto, es decir, usar la versión de una sola línea solo si la operación es sencilla. Si la lógica requiere condiciones complejas, se abren los corchetes y se escribe un return explícito para que se pueda leer el código fácilmente.

    Es recomendable usar nombres descriptivos, al ser funciones anónimas que se guardan en variables, es importante que la constante describa bien qué hace la función.

    Por último, no se debe abusar en los eventos del DOM tradicionales. Si se está manejando un evento tradicional de JS plano con addEventListener, a veces es mejor usar una función normal porque el navegador asigna automáticamente el elemento clickeado al this. Con la función flecha se perdería esa característica.

    Sintaxis

                            const sumarFlechaCompleta = (a, b) => { 
                            return a + b; 
                        };

                        const sumarFlechaCorta = (a, b) => a + b;

                        const saludar = () => "Hola!";

                        const duplicar = x => x * 2;

                        const calcular = (a, b, c) => a + b + c;

    Ejemplos

                        const crearUsuario = (nombre, rol) => ({ nombre: nombre, rol: rol });

                    console.log(crearUsuario("María", "Developer")); 
                
                        Const default = () => console.log('Debe iniciar sesión antes de continuar');

    ¿Qué es la deconstrucción de variables?

    Definición

    La deconstrucción de variables o desestructuración (destructuring assignment) es una sintaxis especial que permite desempacar o extraer valores de los arrays o propiedades de los objetos; sirve para asignarlos de forma directa a variables individuales y limpias utilizando solo una línea de código.

    Como nota importante, la deconstrucción no modifica el objeto o array original, simplemente hace una copia de sus datos hacia las nuevas variables.

    Usos

    Uno de los usos principales de la deconstrucción de variables es determinar los valores por defecto, por si el dato no existe o no es proporcionado. Si se intenta extraer una propiedad que no viene del objeto, se le puede asignar un valor de respaldo para que el código no devuelva undefined.

    Otro uso común es cambiar el nombre de las variables (el alias); a veces el objeto tiene un nombre de propiedad no adecuado o incompatible con otra variable del código. Con la deconstrucción se puede renombrar dicha variable usando solo dos puntos (:).

    Sin embargo, su uso estrella es la desestructuración en los argumentos de una función, de ahí su nombre. Gracias a esto, en lugar de pasar un objeto a una función y escribir cada vez objeto.propiedad; de esta forma se puede desestructurar directamente en los paréntesis de los parámetros.

    Por último, permite saltar posiciones en arrays. Si se necesita seleccionar un elemento concreto de una lista, se pueden dejar espacios vacíos separados por comas para indicar que no se quieren seleccionar dichos elementos (solo vale para listas cortas, con listas largas es más trabajoso que usar otros métodos que utilicen el índice).

    Ventajas y Desventajas

    Ventajas
    • Código DRY (Don't Repeat Yourself): es un término común para expresar que el código está limpio, sin repeticiones innecesarias, ya que elimina la repetición masiva de escribir el nombre del objeto padre una y otra vez.
    • Declaratividad: el inicio del código actúa como documentación, así que, al ver la deconstrucción de la variable, se sabe exactamente qué propiedades del objeto va a utilizar esa sección del código. Esto ayuda a anticiparse a las acciones posteriores.
    • Líneas compactas: permite extraer múltiples datos independientes en una sola línea.
    Desventajas
    • Curva de aprendizaje visual: para principiantes, ver estructuras complejas llenas de llaves, dos puntos y valores por defecto combinados puede resultar difícil de asimilar y entender al principio, ralentizando la forma en la que leen el código.
    • Rigidez en los arrays: en los arrays el orden es primordial. Si la API cambia el orden de los elementos del array, la desestructuración romperá la lógica asignando datos a variables equivocadas.

    Buenas prácticas

    Está desaconsejado abusar de la desestructuración anidada profunda. Aunque JavaScript permite crear desestructuraciones con muchas anidaciones y llaves, es muy complicado leerlas. Si el objeto es muy profundo, es mejor desestructurar en pasos separados para que sea más fácil de asimilar.

    Es recomendable combinarlo con el operador Rest(…). Esto sirve para separar un dato del resto del grupo, y que todo lo que esté dentro de Rest sea un objeto con todos los valores que tenga.

    Por último, una buena práctica es usar la desestructuración para intercambiar variables (Swap). En vez de crear variables temporales para cambiar el valor de dos cajas, utilizando arrays es inmediato.

    Sintaxis

    Deconstrucción de objetos:

                            const nombre = {valores};
                        const {valores} = nombre;

    Deconstrucción de arrays:

                            const nombre = [valores];
                        const [valores] = nombre;

    Ejemplos

                        const configuracion = { volumen: 80 }; 
                    const { volumen, tema = "oscuro" } = configuracion;
                
                        const cliente = { id_interno_sistema: 4022 }; 
                    const { id_interno_sistema: idCliente } = cliente;
                
                        const mostrarPerfil = ({ nombre, edad }) => { 
                        return `Usuario: ${nombre}, Edad: ${edad}`; 
                    }; 
                    const juego = { nombre: "Zelda", consola: "Switch", edad: 10 }; 
                    mostrarPerfil(juego);
                
                        const coordenadas = [10, 20, 30]; 
                    const [, , z] = coordenadas;
                
                        let a = 1; 
                    let b = 2; 
                    [a, b] = [b, a];

    ¿Qué hace el operador de extensión en JS?

    Definición

    El operador de extensión u operador spread toma un elemento iterable (un array, un objeto, una cadena de texto) y lo expande y divide en elementos individuales allí donde se necesiten múltiples argumentos, elementos o propiedades.

    Su función principal es extraer las piezas internas de un contenedor para poder duplicarlas, combinarlas o moverlas a otro contenedor de forma limpia.

    Como analogía podemos usar un mazo de cartas. El array u objeto sería el mazo de cartas dentro de su caja, ordenado; el operador spread sacaría las cartas de la caja y las desplegaría todas sobre la mesa de un solo golpe para poder verlas y usarlas.

    Usos

    Uno de los principales usos del operador de extensión es clonar arrays u objetos sin mutar el original, también llamado copia superficial. En JS si se hace listaB = listaA no se crea una lista nueva, sino que se crea un acceso directo a la misma lista. Por lo que, si se modifica listaB, también se modifica listaA. Con el operador spread se rompe ese enlace creando una copia limpia en un nuevo espacio de la memoria, pudiendo modificar las listas de forma independiente.

    También sirve para fusionar y combinar colecciones, comúnmente llamado concatenación. Permite mezclar múltiples arrays u objetos en uno solo, metiendo los elementos en el orden exacto que se quiera. Esto ofrece mucha libertad a la hora de crear nuevos arrays/objetos a partir de otros prehechos.

    Por último, permite pasar un array como argumentos independientes a una función, especialmente útil para funciones matemáticas como Math.max(), ya que no aceptan un array entero como parámetro, solo aceptan números separados por comas. El operador de extensión resuelve eso de manera sencilla, sin necesidad de código largo y complejo.

    Ventajas y Desventajas

    Ventajas
    • Fomenta la inmutabilidad: es una herramienta muy común en entornos como React, ya que permite actualizar estados (añadir elementos a un alista, cambiar propiedades de un usuario) creando un objeto nuevo sin alterar el original.
    • Sintaxis declarativa: sustituye métodos antiguos como .concat(), Object.assign() o bucles for para copiar datos, volviendo el código mucho más limpio y explicativo.
    • Versatilidad: funciona igual tanto para colecciones indexadas (arrays) como para colecciones de llaves (objetos).
    Desventajas
    • Copia superficial (Shallow Copy): el operador spary solo clona el primer nivel de profundidad, por lo que si el objeto tiene otros objetos o arrays anidados dentro, esos sub-objetos seguirán compartiendo la misma referencia en la memoria, provocando que, si se modifican, el original también se verá alterado.
    • Costo de rendimiento en colecciones masivas: si se intenta propagar un array con millones de registros de golpe, se puede saturar la pila de llamadas (call stack) o consumir demasiada memoria RAM.

    Buenas prácticas

    Es recomendable estar atento a la profundidad de los datos. Si se quiere clonar estructuras con datos anidados complejos (respuestas de APIs JSON profundas o parecido), es mejor no usar el operador de extensión para clonar todo el objeto; sino mejor usar métodos de clonación profunda como structuredClone(objetoOriginal).

    No se debe confundir con el anteriormente mencionado “operador rest”. Aunque se escriban igual (…), hacen cosas opuestas. Si se encuentra en el lado del receptor (antes del igual), es el operador rest, que recoge cosas sueltas y las encajona en un array; si está en el lado del emisor (después del igual(dentro de un array o llamada)), es el operador spread, que desempaqueta el contenido.

    Por último, hay que controlar el orden de la fusión. Al fusionar objetos, se debe colocar siempre las propiedades por defecto al principio y el operador spread de las modificaciones al final. Así es seguro que los datos nuevos sobrescriben correctamente a los antiguos.

    Sintaxis

    const nuevoArray = [...arrayOriginal];
    const nuevoObjeto = { ...objetoOriginal };

    Ejemplos

                        const productosOriginales = ["Camisa", "Pantalón"];

                    const productosCopia = [...productosOriginales];
                    productosCopia.push("Zapatos");
                
                        const usuarioBase = { nombre: "Carlos", rol: "User" }; 
                    const permisosExtra = { rol: "Admin", premium: true };

                    const usuarioFinal = { ...usuarioBase, ...permisosExtra };
                
                        const notas = [4.5, 8.9, 9.2, 6.1];
                    const notaMasAlta = Math.max(...notas);

    ¿Qué es la programación orientada a objetos?

    Definición

    La programación orientada a objetos (POO) es el paradigma de programación más utilizado en la industria del software. Se inventó para solucionar un problema del mundo real: en aplicaciones masivas, el código secuencial tradicional se volvía una maraña de funciones sueltas (código espagueti).

    La POO consiste en organizar el código de la aplicación copiando la forma en la que se organiza el mundo real. En la vida real, el entorno no está hecho de funciones sueltas, está hecho de objetos (un coche, una persona, una taza…) que tienen características propias y que saben hacer cosas. Pues el código imita eso.

    Los programas se estructuran alrededor de datos (propiedades) y comportamientos (métodos) unificados en una sola entidad llamada objeto, en vez de estructurarse alrededor de funciones lógicas puras. La POO tiene dos pilares estructurales:

    • La clase: es el plano arquitectónico (blueprint), el molde o la plantilla teórica.
    • El objeto o instancia: es el producto real construido a partir de ese molde.

    Sin embargo, para que un código se considere realmente orientado a objetos debe cumplir con los siguientes principios fundamentales:

    1. Encapsulamiento (ocultación de datos): Es meter dentro de la “cápsula” del objeto los datos y acciones, protegiendo su estado interno del exterior. Así, el exterior no puede modificar los datos directamente, necesita pedirle permiso al objeto a través de métodos seguros (como los getters y setters).
    2. Abstracción (reducir la complejidad): significa ocultar los detalles complejos de la implementación interna y mostrar solo las características esenciales y sencillas que se necesitan para interactuar con el objeto. Así el código solo muestra lo esencial para poder trabajar con el objeto, sin tener que entender toda la lógica detrás de ello.
    3. Herencia (reutilización de código): permite crear una clase nueva (clase hija) basa en una clase existente (clase padre). La clase hija hereda automáticamente todas las propiedades y métodos de la clase padre, evitando duplicar el código y permitiendo especializar el comportamiento.
    4. Polimorfismo: es la capacidad de que diferentes objetos de diferentes tipos respondan al mismo mensaje o comando, pero ejecutando una acción diferente y adaptada a su propia naturaleza.

    Usos

    La programación orientada a objetos es muy utilizada, por ejemplo, en el desarrollo de videojuegos. Cada personaje, enemigo, bala, obstáculo o escenario es un objeto con propiedades (vida, coordenadas, velocidad) y métodos (moverse, disparar, recibirDaño).

    También es muy utilizada en las interfaces de usuario (UI). En el desarrollo web, los botones, formularios y modales se gestionan como componentes (objetos individuales que se pintan y controlan a sí mismos).

    Por último, en la arquitectura de software empresarial sirve para crear sistemas de facturación, pasarelas de pago o gestión de bases de datos donde las entidades del mundo real (la factura, el cliente, la transacción) se modelan directamente en el código.

    Ventajas y Desventajas

    Ventajas
    • Reutilización de código: gracias a la herencia, no es necesario escribir el mismo código en diferentes partes del programa, vale con extender las funcionalidades de las clases existentes.
    • Mantenimiento modular: si hay un error en un sistema, pongamos el de pagos, basta con ir directo a la clase correspondiente (pasarelaPago). Como está encapsulada, se sabe que los cambios ahí difícilmente romperán otro sistema, como podría ser el de login.
    • Escalabilidad: permite que equipos grandes trabajen en paralelo. Un desarrollador puede estar programando la clase de login mientras otro programa la clase de pagos sin pisarse el código.
    Desventajas
    • Mayor curva de aprendizaje: entender conceptos abstractos como el polimorfismo, las interfaces o la mutabilidad de estados requiere más tiempo que aprender programación secuencial, por lo que consume mucho tiempo de aprendizaje.
    • Mayor consumo de recursos: los objetos ocupan más espacio en la memoria RAM que las variables o funciones primitivas simples.
    • Sobre dificultad: es común caer en la tentación de crear estructuras absurdamente complejas con muchas clases heredadas de otras para resolver problemas sencillos que se solucionan con una simple función de tres líneas.

    Buenas prácticas

    Es recomendable usar el principio de responsabilidad única, es decir, una clase debe encargarse de una sola cosa. No se debe crear una clase, por ejemplo, una llamada usuario, y que guarde los datos de perfil, envíe correos electrónicos de marketing y conecte a la base de datos. Es mejor dividir esa lógica en clases separadas.

    Se prefiere la composición sobre la herencia. No es bueno abusar de la herencia profunda, si la cadena pasa de 2 o 3 niveles, el código se vuelve rígido y difícil de cambiar. Es mejor dotas a los objetos de habilidades combinando componentes más pequeños.

    Por último, es mejor mantener las propiedades privadas por defecto. No se debe dejas que cualquier parte del código altere los valores del objeto libremente. Usar métodos públicos como getters y setters para validad los datos antes de guardarlos es lo ideal.

    Ejemplos

                            this.nombre = nombre;

                        calcularSueldo() {

    ¿Qué es una promesa en JS?

    Definición

    Una promesa (promise) es un objeto que representa el resultado futuro (éxito o fracaso) de una operación asíncrona. En internet, muchas tareas no ocurren al instante (pedir datos a un servidor, cargar una imagen pesada…), si JS esperase a que estas tareas terminaran antes de seguir leyendo el código, la página web se congelaría por completo. Las promesas evitan este mismo problema.

    Básicamente, es un contenedor temporal para un valor que aún no se conoce, pero que se resolverá en algún momento. Mientras ese valor llega, el resto del programa puede seguir ejecutándose sin detenerse.

    Una promesa actúa como una máquina de estados y siempre se encuentra en una de las siguientes situaciones:

    1. Pending/pendiente: es el estado inicial. La operación asíncrona aún no ha terminado.
    2. Fulfilled/cumplida: la operación terminó con éxito. La promesa se resolvió correctamente y ya están los datos disponibles.
    3. Rejected/rechazada: la operación falló debido a un error.

    Usos

    Es muy común su uso para peticiones HTTP/AJAX, para extraer o enviar datos a APIs externas mediante fetch o librerías como Axios.

    También se usan para operaciones de archivos (Node.js), en las que se lee, escribe o borra archivos del disco duro (fs.promises). A su vez, también sirve para consultar registros en bases de datos como MongoDB, PostgreSQL…

    Por último, sirve para crear temporizadores complejos para crear retrasos de tiempo o animaciones encadenadas cronológicamente.

    Ventajas y Desventajas

    Ventajas
    • Se evita el uso del Callback Hell: antes de las promesas, para encadenar tareas asíncronas se tenían que meter funciones dentro de funciones dentro de funciones, dificultando la lectura del código en forma de pirámide. Las promesas permiten encadenar en vertical de forma limpia.
    • Control de errores centralizado: con un solo bloque .catch o un try/catch se puede capturar cualquier error que ocurra en una cadena de cinco o seis operaciones asíncronas seguidas.
    • Ejecución en paralelo: herramientas como Promise.all() o Promise.allSettled() permiten lanzar múltiples tareas al mismo tiempo, multiplicando la velocidad de la aplicación.
    Desventajas
    • Son incancelables: una vez se crea una promesa y se empieza a ejecutar, no se puede detener a mitad de camino de forma nativa (a menos que se usen herramientas avanzadas de controladores como AbortController, pero de por sí no es posible).
    • Tragado silencioso de errores: si se olvida poner un .catch() o un bloque try/catch, la promesa puede fallar en silencio sin que aparezca en el navegador de forma obvia que se rompió, dificultando el debugging.

    Buenas prácticas

    Lo más recomendado es usar async/await por encima de .then(), hace que el código parezca síncrono, eliminando las funciones flechas anidadas y mejorando drásticamente la lectura.

    Es importante no olvidarse del control de errores. No se debe dejar un fetch o una promesa suelta sin su respectivo catch. Una promesa rota sin controlar puede tirar por tierra procesos enteros en entornos como Node.js.

    Cuando sea posible es mejor paralelizar, es decir, no usar await línea por línea para tareas independientes. Si se necesitan pedir datos de tres cosas diferentes que no dependen entre sí, es mejor meterlas en un array y usar Promise.allSettled().

    Por último, es recomendable evitar caer en el “promise hell”, lo cual se traduce en escribir cosas como await api().then(res => res.json().then(data => ...)), es decir, lo mismo que hacían los antiguos callbacks. Es mejor mantener la desestructuración de pasos limpia: un await para la red y otro para el JSON.

    Sintaxis

    Creación de una promesa:

                            const miPromesa = new Promise((resolve, reject) => { 
                            let todoOk = true; 

                            if (todoOk) { 
                                resolve("¡Datos enviados con éxito!"); 
                            } else { 
                                reject("Error: No se pudo procesar."); 
                            }
                        });

    Consumo clásico de una promesa:

                            miPromesa 
                            .then(resultado => console.log(resultado)) 
                            .catch(error => console.error(error)) 
                            .finally(() => console.log("Terminado"));

    Consumo moderno de una promesa:

                            async function ejecutarTarea() { 
                            try { 
                                const resultado = await miPromesa
                                console.log(resultado); 
                            } catch (error) { 
                                console.error(error); 
                            } 
                        }

    Ejemplos

                        const descargarArchivo = () => {
                        return new Promise((resolve) => {
                            console.log("Iniciando descarga...");
                            setTimeout(() => {
                                resolve("Archivo_Secreto.pdf descargado.");
                            }, 3000); 
                        });
                    };

                    async function iniciarApp() {
                        const archivo = await descargarArchivo();
                        console.log(archivo); // Aparece exactamente a los 3 segundos
                    }
                    iniciarApp();

    ¿Qué hacen async y await por nosotros?

    Definición

    async y await don dos palabras clave que trabajan en equipo para permitir escribir código asíncrono que visualmente se lee y se comporta igual que el código asíncrono tradicional (de arriba abajo, línea por línea).

    • async o Asynchronus se coloca antes de la declaración de una función. Su único trabajo es transformar esa función para que siempre retorne una promesa de forma automática, sin importar lo que tenga dentro.
    • await o Wait solo se puede usar dentro de una función marcada como async. Su trabajo es pausar la ejecución de la función en esa línea exacta, esperando a que la promesa se resuelva (tenga éxito o falle) antes de continuar con la siguiente línea. Mientras espera, no congela el navegador, sino que permite que el resto de la aplicación siga funcionando de fondo.

    Usos

    Es muy común encontrarlo en el consumo de APIs y servicios web. Es el estándar para usar fetch o Axios en el desarrollo moderno.

    Permite crear scripts secuenciales, es decir, tareas donde el paso B necesita obligatoriamente los datos que devuelve el paso A.

    Por último, se utiliza mucho en las pruebas unitarias (testing), en herramientas como Jest o Cypress para simular interacciones de usuarios y esperar a que el software responda antes de validar el test.

    Ventajas y Desventajas

    Ventajas
    • Legibilidad imbatible: transforma el código asíncrono en algo lineal. Es mucho más fácil de comprender para el desarrollador.
    • Depuración (debugging) sencilla: si se pone un punto de interrupción (breakpoint) en el navegador para revisar el código línea por línea, con async/await se puede saltar de una línea a otra exactamente en el orden visual en que están escritas. En cambio, con .then() el depurador salta de forma caótica.
    • Estructura de errores nativa: permite usar bloques try/catch. Ya no se necesita aprender métodos raros para capturar fallos, se usa la misma estructura que para el resto del lenguaje.
    Desventajas
    • Peligro de cuellos de botella accidentales: si se pone await en tareas que son independientes entre sí, se obliga a la aplicación a ir más lenta, esperando a procesos que podrían ejecutarse en paralelo.
    • Falsa sensación de sincronismo: al verse tan ordenado, es fácil olvidar que, en el fondo, siguen ocurriendo procesos asíncronos en segundo plano, lo que puede causar confusiones con los tiempos de respuesta si no se comprende bien el funcionamiento del Event Loop de JS.

    Buenas prácticas

    Es importante no olvidarse del try/catch, dejar un await sin protección dentro de una función es dejar una puerta abierta a que la aplicación se congele o se rompa si el servidor externo falla.

    Una buena práctica es identificar qué puede ir en paralelo. Si hay tres peticiones await seguidas que no dependen la una de la otra, es mejor guardar las promesas en variables y combinarlas usando Promise.all() o Promise.allSettled().

    Por último, no se debe abusar de async si no hay await dentro. Es mejor no desgastar recursos marcando todas las funciones del código como async por si acaso; preferiblemente usarse solo si en el cuerpo de la función se van a consumir promesas o se necesita obligatoriamente que devuelva una promesa de forma nativa.

                                const usuario = await res.json();
                                console.log(usuario);
                            } catch (err) {
                                console.error(err); 
                            }
                        }
